## Table 21 — URL読み込み機能仕様（Table 6 STEP 1 追記）

### 概要
STEP 1b「インサイト強化」セクションで、外部URLを入力してテキストを自動抽出し、
見込み客のインサイト情報としてAI生成プロンプトに組み込む。

### 対応URLタイプ
| 種別 | 説明 | 主な用途 |
|---|---|---|
| アンケートURL | GoogleフォームやTypeformの共有リンク | 事前・事後アンケートの回答テキスト |
| YouTube URL | youtube.com / youtu.be | 動画説明文・タイトル（字幕APIは将来対応） |
| 公開ページ | LP・ブログ・Notion公開ページ等 | ターゲット層の言葉・悩みの抽出 |

### 処理フロー
```
1. ユーザーがURL入力 → 「読込」ボタンタップ
2. Next.js API Route（サーバー側）がURLをfetch
3. HTML→テキスト変換（script/style除去・本文テキストのみ抽出）
4. 抽出テキストを「本音インサイト」フィールドに自動補完（上書き確認あり）
5. ユーザーが内容を確認・編集してから次のSTEPへ
6. 抽出テキストをDBには保存しない（セッション内のみ保持）
```

### 制限
- 1生成あたり最大3URL
- 1URLあたりタイムアウト5秒
- 1URLあたり抽出テキスト上限 10,000文字（超過分はトリム）
- `http` / `https` スキームのみ許可

## Table 22 — セキュリティ要件：URL読み込み機能

### SSRF（Server-Side Request Forgery）対策
- URLの事前検証を必須とする
- 以下のIPレンジへのアクセスをブロック：
  - `127.0.0.0/8`（ループバック）
  - `10.0.0.0/8`（プライベートA）
  - `172.16.0.0/12`（プライベートB）
  - `192.168.0.0/16`（プライベートC）
  - `169.254.0.0/16`（リンクローカル）
  - `::1`（IPv6ループバック）
- DNS解決後のIPアドレスも上記チェック対象とする

### スクレイプ時のコンテンツ安全対策
- サーバー側のみでfetch実行（クライアントからの直接アクセス禁止）
- JavaScriptの実行なし（ヘッドレスブラウザ不使用）
- `<script>` `<style>` `<iframe>` タグを除去後にテキスト抽出
- 抽出テキストをAIプロンプトに渡す前にエスケープ処理

### データの取り扱い
- 入力URLはSupabase `user_settings` または `generation_inputs` テーブルにRLS適用で保存
- 抽出済みテキストコンテンツはDBに保存しない（プロンプト生成後に破棄）
- ユーザーが入力したURLは他ユーザーから参照不可（RLSで行レベル制御）

### レート制限
- URL読み込みは1ユーザーあたり1日最大20回
- 超過時は「本日の読み込み上限に達しました」を表示し、手動テキスト入力に誘導

## Table 25 — データ構造定義（Lark管理ファイル一覧）

> 2026-04-05 追記。システムが参照するLarkスプレッドシートおよびその役割を定義する。

### 概要
システム設定データをLarkスプレッドシートで外部管理し、コード変更なしに生成品質・レイアウト・フェーズ設定を調整可能にする。

### データ種別と役割

| ファイル名 | Lark場所 | 種別 | 役割 |
|---|---|---|---|
| 01_レイアウト定義 | ストーリーズ生成AI直下 | システム設定 | 10種のビジュアルレイアウト仕様（id・overlay・text_position等17列） |
| 02_フェーズ設定 | 01.言語化構成チーム | システム設定 | 3フェーズの日数比率・禁止ワード・CTA種別・訴求軸 |
| 03_プロンプトルール | 01.言語化構成チーム | システム設定 | Claudeへの指示ルール集（DO/DONT/良い例/悪い例 14ルール） |
| サキココ① ② ③ ④ | 01.言語化構成チーム | 運用データ | 特定ユーザーの30日分ストーリーズ構成・コンテンツ管理 |

### システム設定 vs 運用データの違い

- **システム設定層**（01〜03）：全ユーザー共通のルール定義。AIの動作を制御する。更新頻度は低く、管理者のみが変更する。
- **運用データ層**（サキココ等）：特定ユーザーの生成結果・投稿スケジュール管理。チームメンバーが日次〜週次で更新する。

### データフロー

```
【システム設定層】
02_フェーズ設定 ─┐
03_プロンプトルール ─┤→ server.js 起動時に読込 → Claude生成に反映
01_レイアウト定義 ──┘（フロントエンドのStoryCanvasが参照）

【運用データ層】
サキココ①②③④ → 生成結果の管理・投稿スケジュール管理（AIの入力ではない）
```

### 詳細説明ドキュメント
各データの詳細な存在意義・更新ルールは、Lark「01.言語化構成チーム」フォルダ内の
「【システム設計】各データの存在意義」ドキュメントを参照すること。
（注：Lark doc作成権限の制約により、内容はローカルの `doc_content.md` にも保存済み）
