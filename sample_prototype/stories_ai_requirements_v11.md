## Table 21 — URL読み込み機能仕様（Table 6 STEP 1 追記）

### 概要
STEP 1b「インサイト強化」セクションで、外部URLを入力してテキストを自動抽出し、
見込み客のインサイト情報としてAI生成プロンプトに組み込む。

### 対応URLタイプ
| 種別 | 説明 | 主な用途 |
|---|---|---|
| アンケートURL | GoogleフォームやTypeformの共有リンク | 事前・事後アンケートの回答テキスト |
| YouTube URL | youtube.com / youtu.be | 動画説明文・タイトル（字幕APIは将来対応） |
| 公開ページ | LP・ブログ・Notion公開ページ等 | ターゲット層の言葉・悩みの抽出 |

### 処理フロー
```
1. ユーザーがURL入力 → 「読込」ボタンタップ
2. バックエンドAPIがURLをfetch（Node.js https モジュール使用）
3. HTML→テキスト変換（script/style/iframe除去・本文テキストのみ抽出）
4. 抽出テキストを「本音インサイト」フィールドに自動補完（上書き確認あり）
5. ユーザーが内容を確認・編集してから次のSTEPへ
6. 抽出テキストをDBには保存しない（セッション内のみ保持）
```

### 制限
- 1生成あたり最大3URL
- 1URLあたりタイムアウト5秒
- 1URLあたり抽出テキスト上限 10,000文字（超過分はトリム）
- `http` / `https` スキームのみ許可

## Table 22 — セキュリティ要件：URL読み込み機能

### SSRF（Server-Side Request Forgery）対策
- URLの事前検証を必須とする
- 以下のIPレンジへのアクセスをブロック：
  - `127.0.0.0/8`（ループバック）
  - `10.0.0.0/8`（プライベートA）
  - `172.16.0.0/12`（プライベートB）
  - `192.168.0.0/16`（プライベートC）
  - `169.254.0.0/16`（リンクローカル）
  - `::1`（IPv6ループバック）
- DNS解決後のIPアドレスも上記チェック対象とする

### スクレイプ時のコンテンツ安全対策
- サーバー側のみでfetch実行（クライアントからの直接アクセス禁止）
- JavaScriptの実行なし（ヘッドレスブラウザ不使用）
- `<script>` `<style>` `<iframe>` タグを除去後にテキスト抽出
- 抽出テキストをAIプロンプトに渡す前にエスケープ処理

### レート制限
- URL読み込みは1ユーザーあたり1日最大20回

## Table 23 — 利用規約追加条項

### 第X条　外部URL読み込み機能の利用

1. **ユーザーの責任**：適法な権限を有するURLのみ使用
2. **禁止事項**：非公開情報・アクセス制限コンテンツの取得・DoS行為
3. **当社の免責**：取得コンテンツの著作権はユーザー責任
4. **コンテンツの取り扱い**：抽出テキストはサーバーに永続保存しない

### 第Y条　AIによる生成コンテンツ

1. 生成コンテンツの正確性・適法性を保証しない
2. 投稿前にユーザー自身が内容を確認する義務
3. 生成コンテンツの著作権はユーザーに帰属

## Table 25 — データ構造定義（Lark管理ファイル一覧）

### 概要
システム設定データをLarkスプレッドシートで外部管理し、コード変更なしに
生成品質・レイアウト・フェーズ設定を調整可能にする。

### データ種別と役割

| ファイル名 | Lark場所 | 種別 | 役割 |
|---|---|---|---|
| 01_レイアウト定義 | ストーリーズ生成AI直下 | システム設定 | 10種のビジュアルレイアウト仕様（17列） |
| 02_フェーズ設定 | 01.言語化構成チーム | システム設定 | 3フェーズの日数比率・禁止ワード・CTA種別 |
| 03_プロンプトルール | 01.言語化構成チーム | システム設定 | Claudeへの指示ルール集（14ルール） |
| 【システム設計】各データの存在意義 | 01.言語化構成チーム | 説明書 | 本表の詳細版・更新ルール |
| サキココ① ② ③ ④ | 01.言語化構成チーム | 運用データ | 特定ユーザーの30日分ストーリーズ構成 |

### システム設定 vs 運用データの違い

- **システム設定層**（01〜03）：全ユーザー共通のルール。AIの動作を制御する。管理者のみ変更。
- **運用データ層**（サキココ等）：ユーザーごとの生成結果・投稿スケジュール管理。

### データフロー

```
システム設定層
  02_フェーズ設定 ─┐
  03_プロンプトルール ─┤→ server.js が読込 → Claude 生成に反映
  01_レイアウト定義 ──┘→ フロントエンド（App.jsx）が参照

運用データ層
  サキココ①②③④ → 生成結果の管理・投稿スケジュール（AIの入力ではない）
```

### Phase 0 追加実装項目

9. Larkスプレッドシート「01_レイアウト定義」作成・10レイアウト定義済み
10. Larkスプレッドシート「02_フェーズ設定」作成・3フェーズ設定定義済み
11. Larkスプレッドシート「03_プロンプトルール」作成・14ルール定義済み
12. 生成中画面（GeneratingScreen）のリアルタイム進捗表示を実装
    - プログレスバー右に%を常時表示
    - Day X / Y 表示（何日目を生成中か）
    - 再試行ボタンがAPIを正しく再実行するよう修正（useEffect + retryCount方式）

## Table 26 — 現行実装の技術構成（MVP開発フェーズ）

|---|---|---|

### バックエンド（現在の実装）

| 項目 | v10仕様 | v11現行実装 |
|---|---|---|
| フレームワーク | Next.js API Routes | Express.js（port 3001） |

### Claude Code CLI 使用方針
- **理由：** 開発初期フェーズでAPIクレジットを使わずに生成機能を検証するため
- **方法：** `spawn("claude", ["-p", "-"])` でstdinにプロンプトを渡す
- **注意：** `ANTHROPIC_API_KEY` をsubprocess環境から除外（Claude Code自体の認証を使用）
- **本番移行時：** Anthropic Claude API（Streaming対応）に切り替え

## Table 27 — フェーズ別生成ルール（AIプロンプト設計）

> server.jsのsystem promptに実装済みのルール定義。Lark「03_プロンプトルール」シートが正式管理場所。

### フェーズ構成

| フェーズ | 期間（30日の場合） | 目的 |
|---|---|---|
| 認知・関係構築 | Day 1〜10（約33%） | 人柄を見せてファンを作る |
| 共感教育 | Day 11〜23（約44%） | 問題の原因を再定義して気づかせる |
| 告知・行動 | Day 24〜30（約23%） | 自然な流れで商品・申込みに誘導 |

### フェーズ別の絶対ルール

**認知・関係構築フェーズ（序盤）**
- 商品・サービス名・個別相談・講座などの告知ワードを一切出さない
- 絶対NG: 「個別相談」「申込」「LINE」「DM」「募集」
- CTAは「アンケートに答えて」「リアクションボタン押して」など超低ハードル

**共感教育フェーズ（中盤）**
- においわせ程度に「こんな方法があります」と示唆するが詳細は出さない
- CTAは「続きが気になる人は保存して」「明日も見てね」など

**告知・行動フェーズ（終盤のみ）**
- ここで初めて具体的な商品・サービス・申込みに言及
- CTAに「DMで『相談』と送って」など具体的アクションを入れる

### 共通ルール

| 項目 | ルール |
|---|---|
| 見出し | 15文字以内・インパクト重視 |
| 本文 | 2〜3文・口語体・改行で読みやすく |
| 出力形式 | JSON配列のみ（説明文・前置き不要） |
| テンション | カジュアルを基本に、告知フェーズのみ上げる |

### 良い例・悪い例

| 区分 | 例 |
|---|---|
| 良い例（認知フェーズ） | 見出し「片付け苦手な私が変わった話」/ 本文：自己開示・エピソード |
| 悪い例（認知フェーズ） | 見出し「個別相談受付中！」← 序盤NG |
| 悪い例（共通） | 「ストーリーズ見てくれてる人いますか？」← 不自然 |
